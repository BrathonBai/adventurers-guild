# 从企业 AI 到企业多 Agent：真正难的不是让 Agent 说话

过去一年，很多企业都在谈 AI。

一开始，大家的想象很直接：给员工配一个 AI 助手，帮他写文档、查资料、做总结、生成周报。这个阶段当然有价值，但很快会碰到天花板。

因为企业不是由一个个孤立的人组成的。

企业真正复杂的地方，不在某个人写一份材料慢了十分钟，而在于一个客户问题要穿过客服、销售、产品、研发、法务、财务；一个事故要穿过值班工程师、服务负责人、客户经理和管理层；一个采购要穿过需求方、审批人、供应商、合同、预算和付款。

这些流程里，最耗时间的往往不是“做事”，而是协调。

谁来接？谁有权限？现在卡在哪？下一步谁审批？这个回复能不能发给客户？这个字段能不能写回系统？如果出了问题，谁负责？

这也是为什么，企业 AI 的下一步不会只是“更聪明的聊天机器人”，而会走向企业多 Agent 架构。

## 单点 AI 的尽头

单点 AI 像一个很聪明的实习生。

你给它材料，它能总结。你让它写邮件，它能起草。你让它分析数据，它能给建议。

但企业里的真实工作，通常不是一个人拿到完整信息后独立完成。真实工作更像一张网：信息在不同系统里，权限在不同角色手里，决策链条有轻重缓急，结果还要回写到业务系统。

所以，一个孤立的 AI 助手很难回答这些问题：

- 它代表谁说话？
- 它能不能访问这份数据？
- 它能不能给客户承诺交付时间？
- 它能不能改 Jira、ServiceNow、CRM、ERP？
- 它做错了，谁能追溯？
- 它和另一个 Agent 的分工由谁决定？

这不是模型聪不聪明的问题，而是组织结构的问题。

企业 AI 如果只停留在个人助手层，提升的是局部效率；企业多 Agent 要解决的，是组织协作效率。

## A2A 很重要，但它不是终点

当企业里出现越来越多 Agent，一个自然的问题就来了：它们怎么互相通信？

Google 提出的 A2A，也就是 Agent-to-Agent 协议，给了一个很好的方向。Agent 可以发现彼此，声明能力，发送任务，交换消息，返回结果。不同厂商、不同运行时、不同部门的 Agent，不必都关在同一个系统里。

这一步非常重要。

但如果把 A2A 理解成“Agent 之间能聊天”，那就低估了企业场景的复杂度。

企业里真正危险的，不是 Agent 不会说话，而是 Agent 说了一句不该说的话，做了一个不该做的动作，或者在没有授权的情况下替某个部门作了承诺。

比如，一个客户事故正在升级。

客服 Agent 想让客户回复 Agent 发一条外部更新：“我们会在周五前修复。”

这句话看起来只是消息，实际上是承诺。它可能影响客户关系、SLA、合同责任，甚至财务赔偿。

所以企业里的 Agent 通信不能只问：“这条消息发出去了吗？”

它还要问：

- 这个 Agent 代表谁？
- 谁授权它这么做？
- 这是建议、草稿，还是正式承诺？
- 是否需要人工审批？
- 涉及哪个客户、哪个工单、哪个业务系统？
- 这次动作有没有审计记录？

A2A 解决的是 Agent 之间的路能不能通。企业还需要交通规则、调度中心和事故追责机制。

## 真正的核心是 Work Graph

很多 AI 产品喜欢围绕对话设计。

但企业不是靠对话运行的，企业靠工作对象运行。

客户、工单、合同、事故、审批、订单、项目、代码变更、销售机会，这些才是企业的真实骨架。

所以，多 Agent 架构要从 Chat 转向 Work Graph。

每一次 Agent 交互，都应该绑定到一个具体工作对象：

这条消息属于哪个工单？这次建议影响哪个审批？这个草稿准备写回哪个客户记录？这个分析基于哪些证据？这个动作完成后，哪个系统状态会改变？

一旦有了 Work Graph，Agent 不再是在聊天窗口里漂浮，而是进入企业的真实业务现场。

它知道自己正在处理什么，和谁协作，下一步要推动什么，以及哪些动作不能越界。

## 授权比权限更重要

传统企业系统谈权限，通常是“这个账号能访问什么”。

但 Agent 时代，仅仅有权限是不够的。

因为 Agent 不是普通账号。它可能代表一个员工、一个团队、一个部门，甚至多个系统自动执行动作。

所以我们需要的是 Delegation，也就是委托授权。

不是简单地说“这个 Agent 能访问 CRM”，而是说：

客服经理授权 Customer Response Agent 在 30 天内读取 Zendesk 和 Salesforce，可草拟客户回复，但不能直接发送；P1 客户事故必须人工审批；涉及退款、赔偿、合同承诺时禁止自动处理。

这才是企业多 Agent 的最小安全单位。

权限回答“能不能进门”，授权回答“进门后能代表谁做什么”。

## 固定流程会变成临时小队

企业软件过去很喜欢 workflow：A 节点结束，流到 B；B 审完，流到 C。

这种模式适合稳定流程，但不太适合复杂问题。

真实问题往往需要临时组队。

一个客户事故来了，可能需要客服负责人、客户经理、研发负责人、知识检索 Agent、日志分析 Agent、客户回复 Agent 一起处理。另一个采购审批来了，需要需求方、预算负责人、法务 Agent、供应商资料 Agent、合同审查 Agent。

这不是固定流程，而是 Party Formation。

任务来了，系统判断它需要哪些能力，哪些人和 Agent 有资格参与，哪些动作需要审批，然后自动组成一个临时执行小队。

这比传统 workflow 更接近企业真实工作方式。

## Connector 不是 API 调用，而是治理边界

Agent 迟早要连接企业系统。

Jira、ServiceNow、Salesforce、Slack、Teams、ERP、知识库、代码仓库，这些系统里的数据和动作才是企业日常。

但连接器不能只是“把 API 接上”。

它必须知道：

- 哪些字段能读？
- 哪些字段能写？
- 哪些动作必须审批？
- 哪些动作永远禁止？
- 写回失败怎么办？
- 原系统和 Agent 平台状态冲突时，以谁为准？

这就是 Connector Governance。

很多企业 AI 项目会死在这里。Demo 阶段看起来很顺，一接真实系统就开始害怕，因为没有明确治理边界。

企业多 Agent 架构如果要落地，连接器必须从第一天就带着治理规则设计。

## 审计之后，还要有信誉

企业当然需要日志。

但只留日志不够。

如果一个 Agent 长期表现稳定，建议经常被采纳，任务很少返工，审批通过率高，它应该获得更高信任。如果另一个 Agent 经常越权、误判、需要人工修正，它就应该被限制在更低自治等级。

这就是 Reputation。

未来企业用 Agent，不应该只看“这个 Agent 会什么”，还要看“这个 Agent 靠不靠谱”。

它的成功率、返工率、审批驳回率、SLA 表现、错误类型、越权尝试，都应该成为调度依据。

企业会慢慢从“给 Agent 分配任务”，走向“根据任务风险和 Agent 信誉自动匹配”。

## 自治不是一次打开的开关

很多人讨论 Agent 自动化时，会在两个极端之间摇摆。

要么完全不信任，所有事情都要人确认；要么过度乐观，想让 Agent 全自动跑完流程。

更现实的做法是风险分层自治。

一开始，Agent 只能读、总结、推荐、草拟。然后，在低风险场景里允许它内部写入，比如更新工单标签、补充内部备注、创建草稿任务。再往后，表现稳定的 Agent 可以在明确授权下自动执行部分动作。

高风险动作仍然保留人工审批。

比如外部客户承诺、生产变更、付款、合同、法律、安全事件，这些不应该轻易交给 Agent 自治。

真正成熟的企业多 Agent 系统，不是让 Agent 一夜之间接管公司，而是让自治能力一点点长出来。

## 从企业 AI 到智能组织层

如果把这条路拉长，会看到一个很清楚的演进路径。

第一阶段，是 Enterprise AI。每个人有自己的 AI 助手，提升个人效率。

第二阶段，是 Department Agents。部门内部有自己的 Agent，处理客服、研发、财务、运营等局部流程。

第三阶段，是 A2A Multi-Agent。不同 Agent 能互相发现、通信、协作。

第四阶段，是 Governed Agent Network。有授权、策略、审批、审计和信誉。

第五阶段，是 Agentic Work Graph。Agent 围绕企业工作图谱自动组队、推进、交付。

第六阶段，是 Adaptive Enterprise。系统能看见组织哪里堵，哪个 Agent 靠谱，哪条流程应该改，哪些授权太宽或太窄，哪些任务可以进一步自动化。

走到这一步，AI 就不再只是工具，而开始变成企业的智能组织层。

## 最后

企业 AI 的难点，从来不只是模型能力。

模型会越来越强，Agent 也会越来越多。真正的问题是：当企业里出现大量 Agent，它们如何被发现、如何协作、如何授权、如何审批、如何审计、如何承担后果。

A2A 是必要的一步，但不是最后一步。

未来真正有价值的企业多 Agent 架构，应该是：

A2A 负责通信，Work Graph 负责业务上下文，Delegation 负责授权，Policy 负责边界，Party Formation 负责组队，Connector Governance 负责系统写回，Audit 和 Reputation 负责追溯与信任，Risk-tiered Autonomy 负责逐步放权。

这套东西合起来，才不是一个聊天机器人，也不是一堆自动化脚本。

它更像企业的新操作系统。
